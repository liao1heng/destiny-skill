# CLI PM

把模糊需求整理成可以直接交付给设计、前端、后端、测试的 PM 文档。

## Core Position

- 这是文档技能，不是实现技能。
- 命中 `cli-pm` 后，只处理文档，不写代码。
- 对页面类需求，默认先写 `.md` 原型图。
- 页面原型图只交付最终页面结构，不交付解释型说明。
- 只有用户明确要求时，才补页面元素规格、逻辑文档、流程文档。

## Hard Guardrails

- 只允许修改文档文件，例如：
  - `doc/**`
  - `docs/**`
  - `prd/**`
  - `spec/**`
  - 用户明确点名的文档型 `readme.txt`、`README.md`、`README.txt`
- 禁止修改源码、测试、配置、接口、样式、脚本、构建、部署文件。
- 用户说“实现 / implement / 落地”时，在本技能下只理解为“把方案落实成文档”。
- 如果用户既要 PM 文档又要代码，先完成文档，再让用户明确切出 `cli-pm`。
- 结束前必须自查：本回合改动是否全部落在文档范围内。

## Default Output Type

### 页面类需求

默认输出：

- `<页面名>-原型图.md`

只在用户明确要求时再补：

- `<页面名>-页面元素.txt`
- `<页面名>-逻辑与规则.txt`

### 流程类需求

- `<流程名>.txt`

### 信息架构类需求

- `<结构名>.txt`

## Prototype Rule

页面原型图必须是 Markdown 文件。

### 原型图要写什么

- 页面顺序
- 每个模块最终显示什么
- 标题
- 副标题
- 卡片
- 按钮文案
- 按钮去向
- 页面不出现什么

### 原型图不要写什么

- 不写模块职责
- 不写“解释 / 说明 / 用于 / 负责”
- 不写页面逻辑分析
- 不写信息层级分析
- 不写 PM 内部备注
- 不写接口教学

### 原型图格式

- Markdown 标题
- 扁平列表
- Markdown 表格
- fenced code block 里的线框结构

不要求画得像设计稿，但必须一眼看出页面最终结构。

## When To Add Extra Docs

只有在以下情况再补别的文档：

- 页面交互很多，测试需要精确元素清单
- 页面存在复杂状态或权限差异
- 页面只是某个跨页面流程里的一个节点
- 用户明确要求“元素文档”和“逻辑文档”分开

## Homepage / Brand Page Anti-Patterns

如果当前是首页、品牌页或营销页，写完后必须自查：

- 有没有把 API、字段、步骤塞进首屏
- 有没有把 PM 规则写成页面文案
- 有没有用内部标签词替代页面内容
- 有没有把文档、状态、支持写成首页主角
- 有没有出现解释型句子，而不是最终页面结果

命中任一项，说明原型图写坏了，要重写。

## What Not To Write

- 不写背景空话。
- 不写“提升体验”“增强可用性”这类泛词。
- 不写内部讨论语、对内策略语、产品经理提醒语。
- 不把首页、品牌页写成功能说明页。
- 不把功能页写成品牌故事。

## Execution Workflow

1. 看仓库现有文档目录和既有文档，优先续写。
2. 如果是页面需求，默认先写 `.md` 原型图。
3. 如有总索引，先更新总索引。
4. 如有必要，再补元素或逻辑文档。
5. 用首页反模式清单自查。
6. 检查改动范围，确保只动文档。
